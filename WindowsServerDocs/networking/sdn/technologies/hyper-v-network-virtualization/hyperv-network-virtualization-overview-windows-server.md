Windows Server 2016 と Virtual Machine Manager では、マイクロソフトはエンド ツー エンドのネットワーク仮想化ソリューションを提供します。  Microsoft のネットワーク仮想化ソリューションを構成する 5 つの主要なコンポーネントがあります。  
  
-   **Windows Azure Pack for Windows Server**テナント接続ポータルを仮想ネットワークを作成し、仮想ネットワークを管理する管理者のポータルを提供します。

-   **HYPER-V ネットワーク仮想化**はネットワーク トラフィックを仮想化するために必要なインフラストラクチャを提供します。  
  
-   **HYPER-V ネットワーク仮想化ゲートウェイ**仮想および物理ネットワーク間の接続を提供します。

## <a name="BKMK_OVER"></a>機能の説明  
HYPER-V ネットワーク仮想化は、「仮想ネットワーク」(VM ネットワークと呼ばれます) を仮想マシンと同様に、サーバー仮想化 (ハイパーバイザー) が、オペレーティング システムに「仮想マシン」を提供する方法を提供します。 ネットワーク仮想化では、物理ネットワーク インフラストラクチャから仮想ネットワークを分離し、仮想マシンのプロビジョニングから VLAN および階層的な IP アドレス割り当ての制約を削除します。 この柔軟性により、簡単に IaaS クラウドに移行する顧客で必要なマルチ テナントの分離、セキュリティの要件を維持しながら、インフラストラクチャを管理するには、ホストやデータ センターの管理者の効率的なとサポートの重複する仮想マシン IP アドレスです。  
  
ユーザーはデータ センターをクラウドにシームレスに拡張を求めています。 現在このようなシームレスなハイブリッド クラウド アーキテクチャを行うには技術的な課題があります。 ユーザーの顔の最も大きな難関の 1 つの既存のネットワーク トポロジ (サブネット、IP アドレス、ネットワーク サービス、およびなど。) クラウドでの再利用であり、内部設置型のリソースとそのクラウド リソース間のブリッジングします。  HYPER-V ネットワーク仮想化は、基盤の物理ネットワークから独立した VM ネットワークの概念を説明します。 この 1 つまたは複数の仮想サブネットから成る、VM ネットワークの概念では、仮想ネットワークに接続された仮想マシンの物理ネットワーク内の正確な場所は、仮想ネットワーク トポロジから切り離されます。 その結果、お客様に簡単に移動できますながら仮想サブネットをクラウドに既存のサービスは、作業を続行できるように既存の IP アドレスとクラウド内のトポロジを維持したまま、サブネットの物理的な場所に関係なく。 つまり、HYPER-V ネットワーク仮想化はシームレスなハイブリッド クラウドを使用できます。  
  
ハイブリッド クラウドに加えて、多くの組織が自社のデータ センターを統合することと、内部的にクラウド アーキテクチャの効率性とスケーラビリティのメリットを取得するプライベート クラウドを作成します。 HYPER-V ネットワーク仮想化は、優れた柔軟性と効率性の事業単位の分離することで、プライベート クラウド ネットワーク トポロジを使用 (これにより仮想) によって、実際の物理ネットワーク トポロジからです。 これにより、事業単位は、簡単を互いに分離されながら、内部プライベート クラウドを共有できを既存のネットワーク トポロジを保持する続行できます。 データ センター運用チームは、柔軟に展開し、データ センター運用効率の向上を提供するサーバーを停止せず、全体的なより効果的なデータ センター内のワークロードを動的に移動します。  
  
ワークロード所有者、主な利点は、移動できる、ワークロードの「トポロジ」をクラウドに IP アドレスの変更やアプリケーションを書き直したりなしです。 たとえば、典型的な 3 層 LOB アプリケーションは、フロント エンド層、ビジネス ロジック層、およびデータベース層で構成されます。 ポリシーを使って HYPER-V ネットワーク仮想化、顧客全部または一部をクラウドに 3 つの層のルーティング トポロジとサービス (バーチャル マシン IP アドレスなど) の IP アドレスを維持しながら、アプリケーションを変更するを必要とせずできます。  
  
インフラストラクチャ所有者の場合は、バーチャル マシンの配置の柔軟性により、ネットワークを再構成したり、仮想マシンを変更することがなく、データ センターで任意の場所ワークロードを移動できます。 たとえばできるように、仮想マシン ライブ マイグレーションできますどこからでも、サービス中断することがなくデータ センターでは、HYPER-V ネットワーク仮想化するサブネット間のライブ マイグレーションが有効にします。 以前のライブ マイグレーションは、仮想マシンがある場所を制限する同じサブネットに制限されていました。 サブネット間は、ライブ マイグレーションは、により、管理者は動的なリソース要件、エネルギー効率に基づいてワークロードを統合して、お客様のワークロード稼働時間を中断することがなくインフラストラクチャのメンテナンスにも対応できます。  
  
## <a name="BKMK_APP"></a>実際の適用  
仮想化データ センターのと共に、IT 組織やホスティング プロバイダー (コロケーションしたり物理サーバーのレンタルを行うプロバイダー) を開始しましたを容易に顧客にオンデマンドでのサーバー インスタンスを提供するためにより柔軟な仮想化インフラストラクチャを提供します。 この新しい形態のサービスは Infrastructure as a Service (IaaS) と呼ばれます。 Windows Server 2016 では、企業ユーザーがプライベート クラウドと、IT への切り替えを as a service の運用モデルを構築するためにすべての必要なプラットフォーム機能を提供します。 Windows Server 2016 2016 では、ホスト側はパブリック クラウドを構築し、IaaS ソリューションを顧客に提供することもできます。 HYPER-V ネットワーク仮想化ポリシーを管理するには、Virtual Machine Manager と Windows Azure Pack と組み合わせると、Microsoft は、強力なクラウド ソリューションを提供します。  
  
Windows Server 2016 HYPER-V ネットワーク仮想化は、管理オーバーヘッドが専用 IaaS クラウドを拡張すると、クラウド ホスト側に提供優れた柔軟性とスケーラビリティ リソース使用率の向上を実現するために仮想マシンを管理するために企業が直面を削減するポリシー ベースでソフトウェア制御のネットワーク仮想化を提供します。  
  
持つ異なる組織部門 (専用クラウドの場合) または異なる顧客 (ホステッド クラウドの) 仮想マシンを扱う IaaS のシナリオでは、セキュアな分離が必要です。 今日のソリューションでは、仮想ローカル エリア ネットワーク (Vlan) は、このシナリオでは、重大な欠点もを表示できます。  
  
**Vlan**  
  
現時点では、Vlan は、ほとんどの組織は、アドレス空間の再利用およびテナントの分離を使用するメカニズムです。 VLAN を使用して明示的な挿入されたタグ (VLAN ID)、イーサネット フレーム ヘッダーと分離を強制し、同じ VLAN ID を使用するネットワーク ノードにトラフィックを制限するイーサネット スイッチに依存しています。 Vlan の主な欠点は次のとおりです。  
  
-   動的なデータ センター内の仮想マシンまたは分離境界を移動するたびに、実稼働スイッチの再構成が面倒ため不注意によるシステム停止のリスクが増大します。  
  
-   スケーラビリティに限り、最大 Vlan 数は 4094 があり、標準的なスイッチがせいぜい 1,000 の VLAN Id をサポートします。  
  
-   1 つの VLAN 内のノードの数を制限し、配置は物理的な場所に基づく仮想マシンの制限を単一の IP サブネット内に制限します。 Vlan は、サイト間で展開できる場合でも、同じサブネットに VLAN 全体があります。  
  
**IP アドレスの割り当て**  
  
仮想マシンの IP アドレスの割り当てでは、Vlan の欠点に加えてのような問題が表示されます。  
  
-   データ センター ネットワーク インフラストラクチャの物理的な場所では、仮想マシンの IP アドレスを決定します。 その結果、通常、クラウドに移動する、サービス ワークロードの IP アドレスを変更する必要があります。  
  
-   ファイアウォール規則、リソースの検出、ディレクトリ サービスなどの IP アドレスには、ポリシーが関連付けられています。 IP アドレスを変更するには、関連付けられているすべてのポリシーを更新する必要があります。  
  
-   バーチャル マシンの展開とトラフィックの分離は、トポロジに依存します。  
  
データ センター ネットワーク管理者は、データ センターの物理的なレイアウトを計画、ときに、サブネットが物理的に配置され、ルーティングについて意思決定を行う必要あります。 これらの決定は、特定のサーバーまたはブレードで、データ センター内の特定のラックに接続されているで実行されている仮想マシンに許容される潜在的な IP アドレスに影響を与える IP とイーサネット テクノロジに基づいています。 仮想マシンがプロビジョニングし、データ センター内に配置は、これらの選択肢と IP アドレスに関する制限に従う必要があります。 そのため、標準的な結果は、データ センターの管理者が仮想マシンに新しい IP アドレスを割り当てることです。  
  
この要件に問題は、アドレスであるだけでなくが IP アドレスに関連付けられているセマンティック情報です。 たとえば、1 つのサブネットは、特定のサービスが含まれている可能性があります。 または個別の物理的な場所にあります。 ファイアウォール規則、アクセス制御ポリシー、および IPsec セキュリティ アソシエーションは、通常、IP アドレスに関連付けられます。 IP アドレスの変更を元の IP アドレスに基づくすべてのポリシーを調整する仮想マシンの所有者を強制します。 この再オーバーヘッドが非常に高い多くの企業がのみ新しいサービスを単独でレガシ アプリケーションのまま、クラウドに展開を選択します。  
  
HYPER-V ネットワーク仮想化は、仮想ネットワークを物理ネットワーク インフラストラクチャからユーザーの仮想マシンを分離します。 その結果、ユーザーの仮想マシンを物理 IP アドレスまたは VLAN Id を再構成しなくても、データ センター内の任意の場所のカスタマー仮想マシンをプロビジョニングするデータ センターの管理者を許可しながら、元の IP アドレスを維持することができます。 次のセクションでは、重要な機能をまとめたものです。  
  
## <a name="BKMK_NEW"></a>重要な機能  
主な機能、利点、および Windows Server 2016 で HYPER-V ネットワーク仮想化の機能の一覧を次に示します。  
  
-   **柔軟なワークロード配置 - ネットワークの分離と IP アドレスの再利用 Vlan なし**  
  
    HYPER-V ネットワーク仮想化は、ホスト側自由にワークロードを配置、データ センター内の物理ネットワーク インフラストラクチャからお客様の仮想ネットワークを分離します。 仮想マシンのワークロード配置は、ソフトウェア定義型、マルチ テナントの仮想化ポリシーに基づいて HYPER-V ホスト内で強制されるために不要になった、IP アドレスの割り当てまたは物理ネットワークの VLAN 分離要件によって制限されます。  
  
    面倒な VLAN 構成を必要とするか、IP アドレス階層に反することがなく、同じホスト サーバーで、重複する IP アドレスを持つ別のユーザーからの仮想マシンを展開できるようになりましたできます。 これにより、お客様の作業負荷共有 IaaS ホスティング プロバイダー、加えなければまま変更されていない仮想マシンの IP アドレスが含まれています。 これらのワークロードを移動することも可能に移行が合理化できます。 多数のユーザーは既存のネットワーク アドレス空間を共有 IaaS データ センターを拡張する必要をサポートするため、ホスティング プロバイダーの構成と分離した Vlan 可能性のある重複したアドレス空間の共存できるようにするには、各顧客を維持する複雑な作業です。 HYPER-V ネットワーク仮想化では、重複するアドレスのサポートが容易になり、ホスティング プロバイダーが少ないネットワークの再構成が必要です。  
  
    さらに、物理インフラストラクチャのメンテナンスやアップグレードは、お客様の作業負荷のダウン時間を与えることがなく実行できます。 HYPER-V ネットワーク仮想化では、物理 IP アドレスの変更や大がかりな再構成を必要とせずに特定のホスト、ラック、サブネット、VLAN、またはクラスター全体の仮想マシンを移行できます。  
  
-   **共有 IaaS クラウドへのワークロードの簡単な移動にします。**  
  
    HYPER-V ネットワーク仮想化では、IP アドレスとバーチャル マシンの構成は変更されません。 これにより、IT 組織をより簡単にワークロードをデータ センターからワークロードまたはインフラストラクチャ ツールとポリシーの最小限の再構成の共有 IaaS ホスティング プロバイダーに移動します。 場合も 2 つのデータ センター間の接続では、IT 管理者がそれらを再構成しなくても、これらのツールを使用する続行できます。  
  
-   **サブネット間のライブ マイグレーション**  
  
    仮想マシンのワークロードのライブ マイグレーションが限られていました、同じ IP サブネットまたは VLAN にサブネットをまたぐ、バーチャル マシンのゲスト オペレーティング システムをその IP アドレスを変更するために必要なためです。 このアドレスの変更は、既存の通信し、仮想マシンで実行されているサービスを中断します。 HYPER-V ネットワーク仮想化ワークロードできるライブ ワークロードの IP アドレスを変更することがなく別のサブネットに Windows Server 2016 を実行しているサーバーを 1 つのサブネットで Windows Server 2016 を実行しているサーバーから移行です。 HYPER-V ネットワーク仮想化は、ライブ マイグレーションによる仮想マシンの場所の変更を更新、および移行された仮想マシンと通信中のホスト間で同期されることを確認します。  
  
-   **切り離されるサーバーとネットワークの管理の簡単な管理にします。**  
  
    移行とワークロードの配置が、基になる物理ネットワークの構成に依存しないために、サーバー ワークロードの配置が単純化されます。 サーバー管理者がサービスとサーバーの管理に専念でき、ネットワーク管理者は全体的なネットワーク インフラストラクチャとトラフィック管理に専念できます。 これにより、データ センター サーバー管理者を展開および仮想マシンの IP アドレスを変更することがなく仮想マシンを移行できます。 オーバーヘッドが減りため、HYPER-V ネットワーク仮想化により、ネットワーク トポロジとは独立してバーチャル マシンの配置、分離境界が変わる可能性がある配置で使用するネットワーク管理者の負担が減ります。  
  
-   **ネットワークを単純化およびサーバー/ネットワーク リソース使用率の向上**  
  
    できなかったり、Vlan と物理ネットワーク インフラストラクチャの仮想マシンの配置の依存関係は、縛りとリソースを十分になります。 依存関係を分割することによりバーチャル マシンのワークロードの配置の柔軟性が向上は、ネットワークの管理を簡略化し、サーバーとネットワーク リソース使用率の向上します。 HYPER-V ネットワーク仮想化が物理的なデータ センターのコンテキストで Vlan をサポートしていることに注意してください。 たとえば、データ センターでは、特定の VLAN 上にあるすべての HYPER-V ネットワーク仮想化トラフィックを必要があります。  
  
-   **既存のインフラストラクチャおよび新技術との互換性が**  
  
    HYPER-V ネットワーク仮想化は、今日のデータ センターに展開することができますも出現 datacenter「フラット ネットワーク」テクノロジとの互換性です。  
  
    たとえば、Windows Server 2016 での HNV サポート VXLAN カプセル化形式と Open vSwitch データベース Management Protocol (OVSDB) として、SouthBound Interface (SBI).  
  
-   **相互運用性およびエコシステム準備では、します。**  
  
    HYPER-V ネットワーク仮想化には、クロス プレミス接続、記憶域エリア ネットワーク (SAN)、リソースの仮想化されていないアクセス、およびのように、既存のリソースとの通信に複数の構成がサポートしています。 Microsoft をサポートし、パフォーマンス、スケーラビリティ、および管理容易性の観点から、HYPER-V ネットワーク仮想化のエクスペリエンスの向上、エコシステム パートナーと連携してに努めています。  
  
-   **ポリシー ベースの構成**  
  
    Microsoft ネットワーク コント ローラーでは、Windows Server 2016 でネットワーク仮想化ポリシーが構成されています。 ネットワーク コント ローラーが、RESTful northbound API とポリシーを構成するための Windows PowerShell インターフェイス。 Microsoft ネットワーク コント ローラーの詳細については、次を参照してください。[ネットワーク コント ローラー](../../../sdn/technologies/network-controller/../../../sdn/technologies/network-controller/Network-Controller.md)します。  
  
## <a name="BKMK_SOFT"></a>ソフトウェアの要件  
Microsoft ネットワーク コント ローラーを使用して HYPER-V ネットワーク仮想化では、Windows Server 2016 と HYPER-V の役割が必要です。
