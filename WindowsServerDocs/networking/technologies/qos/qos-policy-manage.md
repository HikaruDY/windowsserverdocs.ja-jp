##  <a name="bkmk_createpolicy"></a>QoS ポリシーを作成します。

QoS ポリシーを作成する前に、ネットワーク トラフィックを管理するための 2 つのキー QoS コントロールを理解している必要があります。

- DSCP 値

-   スロットル率

### <a name="prioritizing-traffic-with-dscp"></a>Dscp のトラフィックの優先順位を付ける

使用して送信ネットワーク トラフィックの優先順位を定義するには述べたように前の基幹業務アプリケーションの例では、**DSCP 値を指定**を特定の DSCP 値で QoS ポリシーを構成します。 

RFC 2474 で説明されている、DSCP 値は 0 ~ 63 および IPv6 の Traffic Class フィールド内の IPv4 パケット TOS フィールドで指定することができます。 ネットワーク ルーターは、ネットワーク パケットを分類し、それらを適切にキューに入れ、DSCP 値を使用します。
  
> [!NOTE]
>  既定では、Windows のトラフィックは、DSCP 値を 0 を持っています。
  
キューの数およびその優先順位付けの動作は、組織の QoS 戦略の一環として設計される必要があります。 たとえば、組織が 5 つのキューに選択可能性があります。待機時間の影響を受けやすいトラフィック、制御トラフィック、ビジネス クリティカル トラフィック、ベストエフォート型トラフィック、および一括データ転送トラフィック。  
  
### <a name="throttling-traffic"></a>トラフィックの調整

DSCP 値と共に調整はネットワーク帯域幅を管理するための別のキー コントロールです。 前述のようは、使用、**スロットル率を指定**QoS ポリシーの送信トラフィックの特定のスロットル率を構成に設定します。 調整を使用して、QoS ポリシーは、指定されたスロットル率を送信ネットワーク トラフィックを制限します。 DSCP のマーキングとスロットルは、トラフィックを効果的に管理する同時使用できます。

>[!NOTE]
>既定で、**スロットル率を指定**] チェック ボックスが選択されていません。

QoS ポリシーを作成するには、グループ ポリシー オブジェクト (GPO) から、グループ ポリシーの管理コンソール (GPMC) ツール内の設定を編集します。 GPMC は、グループ ポリシー オブジェクト エディターを開きます。

QoS ポリシーの名前は一意である必要があります。 サーバーとエンド ユーザーにポリシーを適用する方法は、QoS ポリシーがグループ ポリシー オブジェクト エディターで保存されているかによって異なります。

- コンピューターの構成 \windows Settings\QoS ポリシー内の QoS ポリシーは、現在ログオンしているユーザーに関係なく、コンピューターに適用されます。 通常、サーバー コンピューターには、コンピューター ベースの QoS ポリシーを使用します。

- ユーザーの構成 \windows Settings\QoS ポリシー内の QoS ポリシーは、ログオンしたにどのコンピューターに関係なくログオンした後に、ユーザーに適用されます。

### <a name="advanced-settings-for-roaming-and-remote-users"></a>ローミングとリモート ユーザーの詳細設定  
QoS ポリシーを使用する目的は、企業のネットワーク上のトラフィックを管理します。 モバイル シナリオでは、またはエンタープライズ ネットワークを無効にユーザーがトラフィック送信する可能性があります。 QoS ポリシーが、企業がネットワークから離れているときに関連していないために、Windows 8、Windows 7、または Windows Vista の企業に接続されているネットワーク インターフェイスでのみ QoS ポリシーが有効にします。  
  
たとえば、ユーザーは、コーヒー ショップから、仮想プライベート ネットワーク (VPN) 経由での自分の企業のネットワークに自分のポータブル コンピューターを接続可能性があります。 Vpn では、(ワイヤレス) など、物理ネットワーク インターフェイスを QoS ポリシーを適用する必要はありません。 ただし、VPN インターフェイスは、企業に接続するため、適用する QoS ポリシーがあります。 後で AD DS の信頼関係がない別の企業のネットワークに入ると、ユーザー、QoS ポリシーは有効になりません。  
  
これらのモバイル シナリオ サーバー ワークロードに適用されないことに注意してください。 たとえば、企業のネットワークの端に複数のネットワーク アダプターを持つサーバーに座っている可能性があります。 IT 部門の QoS ポリシーのスロットルのトラフィックには、エンタープライズの egresses ことができます。ただし、この出口トラフィックを送信するネットワーク アダプターは必ずしも接続できませんエンタープライズ ネットワークに戻す。 このため、Windows Server 2012 を実行しているコンピューターのすべてのネットワーク インターフェイスの QoS ポリシーは常に有効にします。  
  
> [!NOTE]

スループットの低下をレベルで設定して、**受信 TCP トラフィック**TCP で制限する] タブで、TCP の受信ウィンドウ サイズが提供します。 この設定の効果は、スループットが増加するしより広い帯域幅と待機時間 (帯域幅の遅延製品) TCP 接続の使用率をリンクします。 既定では、Windows Server 2012、Windows 8、Windows Server 2008 R2、Windows Server 2008、および Windows Vista を実行しているコンピューターは、最大スループット レベルを設定します。
  
TCP の受信ウィンドウが Windows の以前のバージョンから Windows Server 2012、Windows 8、Windows Server 2008 R2、Windows Server 2008、および Windows Vista に変更します。 以前のバージョンの Windows に最大 64 キロバイト (KB) の TCP の受信側ウィンドウが制限されている場合、受信側ウィンドウが動的にサイズ Windows Server 2012、Windows 8、Windows Server 2008 R2、Windows Server 2008、および Windows Vista が、最大 16 メガバイト (MB)。 着信 TCP トラフィック管理で、TCP 受信-ウィンドウを拡張できる最大値を設定して、入力方向のスループット レベルを制御できます。 レベルは、次の最大値に対応します。

実際のウィンドウのサイズと等しいかネットワークの状態に応じて、最大値より小さい値があります。

### <a name="BKMK_precedencerules"></a>QoS ポリシーの優先順位の規則

GPO の優先順位と同様に、QoS ポリシーは、特定のトラフィックのセットに複数の QoS ポリシーを適用するときの競合を解決するのには規則を優先順位を設定します。 送信の TCP または UDP トラフィック時に、QoS ポリシーはスロットル率の合計場所などの累積的な影響がないことを意味する 1 つだけの QoS ポリシーを適用できます。

一般に、最も一致する条件で QoS ポリシーが優先されます。 複数の QoS ポリシーを適用するときに、規則 3 つのカテゴリに分類: レベルですコンピューターとユーザー レベル。ネットワークの組; とアプリケーションネットワーク間で quintuple します。

によって*ネットワーク 5 倍*、ことを意味する送信元 IP アドレス、宛先 IP アドレス、発信元ポート、移行先のポートとプロトコルの \(TCP/UDP\) おします。  

 **ユーザー レベルの QoS ポリシーはコンピューター レベルの QoS ポリシーより優先されます。**

このルールは、特にユーザー ポリシーをグループ ベースの QoS の Gpo のネットワーク管理者の管理が大幅に容易になります。 たとえば、ネットワーク管理者は、ユーザー グループの QoS ポリシーを定義する必要がある場合、だけを作成して配布できます GPO をそのグループに。 必要はありません心配にどのコンピューターについてそれらのユーザーのログオン、かどうかこれらのコンピューターが定義されている、QoS ポリシーの競合しているため、競合が存在する場合、ユーザー レベルのポリシーが常に優先します。

> [!NOTE]
>  ユーザー レベルの QoS ポリシーは、そのユーザーによって生成されるトラフィックに適用されるはのみです。 特定のコンピューターと自体は、コンピューターの他のユーザーは、そのユーザーに対して定義されているすべての QoS ポリシーの対象はできません。

 **アプリケーション特異性とネットワーク組経由で対象よりも優先されます。**

複数の QoS ポリシーには、特定のトラフィックが一致すると、特定のポリシーが適用されます。 アプリケーションを識別するポリシー、間で送信元のアプリケーションのファイルのパスを含むポリシーがのみ、アプリケーション名 (path なし) を識別する別のポリシーよりも具体的と見なされます。 アプリケーションで複数のポリシーが引き続き適用する場合、優先順位の規則は、最良の一致を検索する 5 倍のネットワークを使用します。

または、複数の QoS ポリシーは、重複しない条件を指定することによって同じのトラフィックに適用する可能性があります。 アプリケーションの条件とネットワーク 5 回連続では、間でアプリケーションを指定するポリシーがより具体的と見なされますされ、適用されます。 

たとえば、policy_A は、アプリケーション名 (app.exe) のみを指定し、policy_B が宛先 IP アドレス 192.168.1.0/24 時間を指定します。 これらの QoS ポリシーの競合とき \ (app.exe トラフィックを送信 192.168.4.0/24\ の範囲内の IP アドレスに)、policy_A を適用します。

 **複数特異性が優先ネットワーク内で 5 倍**

ネットワーク 5 回内では、ポリシーの競合は最も一致する条件でポリシーが優先されます。 たとえば、policy_C は発信元 IP アドレス「、」は、宛先 IP 10.0.0.1、発信元ポート「任意」の宛先ポートに「、」対応"TCP"のプロトコルを指定します。 

次に、policy_D 発信元 IP アドレス「、」、宛先 IP 10.0.0.1、発信元ポート「任意」の宛先ポート 80 に対処し、プロトコルの「TCP」を指定すると想定します。 その policy_C と policy_D 宛先 10.0.0.1:80 への接続と一致します。 QoS ポリシーに一致する最も固有の条件でポリシーが適用されるため policy_D はこの例でも優先されます。  
  
ただし、QoS ポリシーは、条件の数が等しい場合があります。 たとえば、いくつかのポリシーが各を指定 1 つだけ (ただし、同じではありません) 1 つのネットワーク 5 回です。 ネットワーク 5 回間で、次の順序はから高い優先順位を下げるには。

- 発信元 IP アドレス

- 宛先 IP アドレス

- 発信元ポート

- 宛先ポート

- プロトコル (TCP または UDP)

IP アドレスなど、特定の条件内では、複数の特定の IP アドレスが優先順位の高い; 扱われますたとえば、IP アドレス 192.168.4.1 は 192.168.4.0/24 よりもより具体的です。
