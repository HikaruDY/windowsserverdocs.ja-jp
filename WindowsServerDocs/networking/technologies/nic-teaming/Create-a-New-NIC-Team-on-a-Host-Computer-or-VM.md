## <a name="bkmk_teaming"></a>チーミング モードを選択します。  
チーミング モードのオプションは、**スイッチに依存しない**、**静的チーミング**、および**コントロール プロトコル LACP (Link Aggregation)**します。 静的チーミングと LACP にはスイッチ依存モードです。 次の 3 つすべてのチーミング モードと最高の NIC チームのパフォーマンスを動的配布の負荷分散モードを使用することをお勧めします。  
  
**スイッチ依存**  
  
スイッチに依存しないモードでは、スイッチまたはスイッチ、NIC チームのメンバーが接続する NIC チームの存在に対応していないし、NIC チームのメンバーのネットワーク トラフィックを分散する方法を決定できませんに代わりに、NIC チームは分散受信ネットワーク トラフィック、NIC チーム メンバーされます。  
  
スイッチに依存しないモード動的配布を使用する場合は、動的負荷分散アルゴリズムによって変更された、TCP ポート アドレスのハッシュに基づいてネットワーク トラフィックの負荷が配布されます。 動的負荷分散アルゴリズムは、個々 のフロー転送 1 つのアクティブなチーム メンバーから別に移動できますできるように、チーム メンバーの帯域幅の使用率を最適化するためにフローを再分配します。 このアルゴリズムでは、まれにいるトラフィックを再配布する原因となる、パケットのアウト オブ オーダー配信のため、その可能性を減らすための手順がかかりますが考慮されます。  
  
**スイッチ依存**  
  
スイッチ依存モードでは、スイッチを NIC チームのメンバーが接続されている NIC チームのメンバー間で受信ネットワーク トラフィックを分散する方法を決定します。 スイッチでは、ネットワーク トラフィックを NIC チーム メンバー間で配布する方法を決定する完全な独立性が存在します。  
  
> [!IMPORTANT]  
> スイッチ依存してチーム化する必要がありますチーム メンバー全員が同じ物理スイッチに接続されているマルチ シャーシ スイッチ間での複数のシャーシ スイッチ ID を共有します。  
  
静的チーミングでは、スイッチとを特定し、ホストの両方を手動で構成する必要がありますリンクは、チームを形成します。 これは静的に構成されたソリューションであるため、スイッチを支援する追加のプロトコルはなく、ホストを正しく特定するには、ケーブルまたはその他のチームの実行に失敗する原因となるエラーが接続されています。 このモードは通常、サーバー クラスのスイッチでサポートされます。  
  
静的チーミング] とは異なり LACP チーミング モードは、ホストとスイッチ間で接続されているリンクを動的に識別します。 この動的な接続により、送信またはピア エンティティから LACP パケットの受信をだけでチームの理論的がほとんどプラクティス、拡張、およびチームの削減に自動的に作成します。 サーバー クラスのすべてのスイッチが LACP をサポートし、すべてを管理、スイッチ ポートの LACP を有効にするネットワーク オペレーターを必要とします。 Lacp チーミング モードを構成するときに NIC チーミングと短いタイマ LACP のアクティブなモードで常に動作します。  オプションは、タイマーを変更または LACP モードを変更できる現在ではありません。  
  
動的配布とスイッチ依存モードを使用すると、ネットワーク トラフィックの負荷は、動的負荷分散アルゴリズムによって変更された TransportPorts アドレスのハッシュに基づく配布されます。  動的負荷分散アルゴリズムは、チーム メンバーの帯域幅の使用率を最適化するためにフローを再分配します。 個々 のフローの転送は、動的配布の一部として別にアクティブな 1 つのチーム メンバーから移動できます。 このアルゴリズムでは、まれにいるトラフィックを再配布する原因となる、パケットのアウト オブ オーダー配信のため、その可能性を減らすための手順がかかりますが考慮されます。  
  
ようにすべてのスイッチ依存する構成では、スイッチは、チーム メンバー間での着信トラフィックを分散する方法を決定します。  スイッチがチーム メンバー間でトラフィックを分散する妥当なジョブを実行すると想定されますが、完全な独立は方法を決定します。  
  
## <a name="bkmk_lb"></a>負荷分散モードを選択します。  
負荷分散の分散モードのオプションは、**アドレスのハッシュ**、 **HYPER-V ポート**、および**動的**します。  
  
**アドレスのハッシュ**  
  
この負荷分散モードでは、パケットのアドレスのコンポーネントに基づいてハッシュを作成します。 これは、後、利用可能なアダプターのいずれかには、そのハッシュ値を持つパケットを割り当てます。 通常このメカニズムだけでは、使用可能なアダプター間で適切なバランスを作成するために十分です。  
  
Windows PowerShell を使用するには、次のハッシュ関数コンポーネントの値を指定します。  
  
-   移行元と移行先の TCP ポートおよび送信元と送信先の IP アドレスをします。 これを選択すると、既定値は、**アドレスのハッシュ**負荷分散モードとします。  
  
-   ソースと宛先 IP アドレスのみです。  
  
-   移行元と移行先の MAC アドレスのみです。  
  
TCP ポート ハッシュより小さなストリームが独立して移動できる NIC チームのメンバー間でトラフィック ストリームの最も詳細な配布を作成します。 ただし、TCP ではないトラフィックに TCP ポートのハッシュを使用することはできませんまたは UDP ベース、ここで TCP および UDP ポートがスタックから隠さなど、トラフィックが IPsec で保護されているか。 このような場合は、ハッシュが自動的に IP アドレスのハッシュを使用してまたは、トラフィックが IP トラフィックでない場合は MAC アドレスのハッシュが使用します。  
  
**HYPER-V ポート**  
  
HYPER-V ホストで構成されている NIC チームの HYPER-V ポートのモードを使用する利点があります。 Vm に独立した MAC アドレスがあるため、VM の MAC アドレスまたは HYPER-V 仮想スイッチ上に VM が接続されているポートは、NIC チーム メンバー間のネットワーク トラフィックを分割する基準を指定できます。  
  
> [!IMPORTANT]  
> HYPER-V ポートの負荷分散モードでは、Vm 内で作成した NIC チームを構成することはできません。 代わりに、使用するアドレスのハッシュで負荷分散モード。  
  
隣接するスイッチは、常に 1 つのポートで特定の MAC アドレスを見て、ため、スイッチは、接続先 MAC (VM の MAC) アドレスに基づく複数のリンク受信負荷 (ホストのスイッチからトラフィック) を配布します。 これは特に便利ですバーチャル マシン キュー (VMQs) を使用する場合、キューは、ここで、トラフィックが到着する期待される特定の NIC で配置できるためです。  
  
ただし、ホストにいくつかの Vm がある場合は、このモードできない可能性がありますな負荷分散を実現するために必要なレベルです。 このモードは、1 つのインターフェイスで利用できる帯域幅を 1 つの VM (つまり、1 つのスイッチ ポートからのトラフィック) を常にも制限されます。 NIC チーミング スイッチ ポートで MAC アドレスを 1 つ以上を持つ場合によっては、バーチャル マシンを構成する可能性がありますので、ソース MAC アドレスを使用する代わりに識別子として、HYPER-V 仮想スイッチ ポートを使用します。  
  
**動的**  
  
この負荷分散モードでは、その他の 2 つのモードのそれぞれの最適な側面は利用し、それらを 1 つのモードにまとめたもの。  
  
-   TCP ポートと IP アドレスのハッシュに基づいて、送信の負荷が分散されます。  動的モードもを再分配負荷をリアルタイムできるように、特定の送信フローはチーム メンバー間で双方向に移動する可能性があります。  
  
-   HYPER-V ポートのモードと同じ方法では、着信負荷が分散されます。  
  
このモードでの送信の負荷が flowlets の概念に基づいて動的に分散します。 人間の音声認識に単語や文の末尾にある自然な改行があると同様 TCP フロー (TCP 通信ストリーム) が自然に発生した中断もあります。 このような 2 つの区切りの間の TCP フローの部分は、"を flowlet と"と呼ばれます。  
  
動的モード アルゴリズムに十分な長さの中断が TCP フロー - で発生した場合など flowlet 境界が検出されたことを検出すると、アルゴリズムで該当する場合、別のチーム メンバーへの流れが自動的に再分散します。  状況によっては、アルゴリズム可能性があります、flowlets が含まれていないフローを再調整も定期的にします。 このため、TCP フローとチーム メンバー間のアフィニティは動的な分散アルゴリズムは、チーム メンバーの作業負荷のバランスをとる動作いつでも変更できます。  
  
チームがスイッチに依存しないまたはスイッチ依存モードのいずれかで構成されているかどうかは、最適なパフォーマンスを動的配布モードを使用することをお勧めします。  
  
この規則の例外がある場合、NIC チームが 2 つのチーム メンバーがスイッチに依存しないモードで構成され、アクティブ/スタンバイ モードが有効になっている、アクティブな 1 つの NIC とおよびその他のスタンバイ用に構成されています。 この NIC チームの構成では、アドレスのハッシュの配布は配布動的のよりも若干優れたパフォーマンスを提供します。  
  
## <a name="bkmk_standby"></a>スタンバイ アダプター設定を選択します。  
スタンバイ アダプターのオプションは、[なし] (すべてのアダプター アクティブ) または他のチームが選択されていないメンバーがアクティブなときに、スタンバイ アダプターとして機能する NIC チーム内の特定のネットワーク アダプターの選択されます。 なしのネットワーク トラフィックに送信するかしない限り、アダプターで処理されたスタンバイ アダプターと NIC を構成するときに、作業中の NIC に失敗すると時点まで。 Active の NIC が失敗した後スタンバイ NIC にアクティブになるし、ネットワーク トラフィックを処理します。 チーム メンバー全員がサービスに復元される場合は、スタンバイのチーム メンバーがスタンバイ状態に返されます。  
  
2 枚の NIC チームがあり、スタンバイ アダプターと NIC が 1 つを構成する場合は失われます、帯域幅集約長所アクティブな 2 つの Nic に存在します。  
  
> [!IMPORTANT]  
> フォールト トレランスを実現するために、スタンバイ アダプターを指定する必要はありません。フォールト トレランスは常に NIC チーム内には、少なくとも 2 つのネットワーク アダプターがある場合は、常に存在します。
