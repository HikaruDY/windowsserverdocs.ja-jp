# <a name="audit-policy-recommendations"></a>監査ポリシーの推奨事項

>適用先:Windows Server 2016、Windows Server 2012 R2、Windows Server 2012、Windows 10、Windows 8.1、Windows 7

このセクションでは Windows 既定の監査ポリシーの設定、ベースラインの推奨される監査ポリシーの設定より積極的な推奨事項、Microsoft からワークステーションおよびサーバー製品。  

管理者が開始基準ガイドにのみ、侵害を検出するために推奨設定と共にここでは、表示、SCM ベースラインの推奨事項はします。 各組織直面している脅威、その許容できるリスクの許容範囲、およびどのような監査ポリシー カテゴリまたはサブカテゴリが有効にする必要がありますに関して独自の意思決定してください。 脅威についてを参照してください、[脅威と対策ガイド](https://technet.microsoft.com/library/hh125921(v=ws.10).aspx)します。 インプレース親切な監査ポリシーのない管理者は、ここでは、推奨設定で開始することをお勧めし、次を変更して、運用環境に実装する前に、テストします。  

推奨事項では、エンタープライズ クラス コンピューターは、Microsoft が平均セキュリティ要件があり、高レベルの運用機能を必要とするコンピューターとして定義します。 高度なセキュリティ要件をより積極的な検討を必要とするエンティティは、ポリシーを監査します。  

> [!NOTE]  
> Microsoft Windows の既定値し、ベースラインの推奨事項で撮影された、 [Microsoft セキュリティ コンプライアンス マネージャー ツール](https://technet.microsoft.com/library/cc677002.aspx)します。  

次の基準監査ポリシーの設定は断固たる敵対者またはマルウェアによって、使用中の成功した攻撃の下にある不明なコンピューターを通常のセキュリティをお勧めします。

## <a name="set-audit-policy-on-workstations-and-servers"></a>ワークステーションとサーバーの監査ポリシーの設定  
すべてのイベント ログ管理の計画には、ワークステーションとサーバーを監視する必要があります。 よくある間違いでは、のみのサーバーまたはドメイン コント ローラーを監視します。 悪意のあるハッキングなどの多くの場合、最初に発生するためのワークステーションで、ワークステーションを監視するいないと情報の最適なと最も古いソースは無視します。  

管理者は熟考のうえを確認し、運用環境に実装する前にすべての監査ポリシーをテストします。  

## <a name="events-to-monitor"></a>監視するイベント  
セキュリティ アラートを生成する完全なイベント ID は、次の属性を含める必要があります。  

-   可能性が高いその発生が許可されていないアクティビティを示します  

-   偽陽性の数が少ないこと  

-   調査またはフォレンジクスの応答に見つかったが発生する必要があります。  

2 種類のイベントを監視および警告を表示する必要があります。  

1.  これらのイベントも 1 回の発生が許可されていないアクティビティを示します  

2.  予想および許容される基準を超えて繰り返し発生するイベント  

最初のイベントの例を示します。  

Domain Admins (DAs) ドメイン コント ローラーではないコンピューターにログオンからすることはできません、エンドユーザーのワークステーションにログオンしている DA メンバーの 1 つには警告を生成して、調査します。 この種類のアラートは 4964 特別なログオンの監査イベントを使用して生成する簡単な (特殊なグループを新しいログオンに割り当てられている)。 他の 1 つのインスタンスのアラートの例は次のとおりです。  

-   サーバー A は、サーバー B で相互に接続するときにアラートを接続しない必要があります。 場合、  

-   通常エンドユーザー アカウントは予期せず、重要なセキュリティ グループに追加する場合のアラートを生成します。  

-   ファクトリの場所、従業員は、夜、アラート午前 0 時にユーザーのログオン時に動作しません。 場合、  

-   ドメイン コント ローラーで、承認されていないサービスがインストールされている場合に警告します。  

-   正規のエンドユーザーが直接 SQL Server があるない明確な理由にログオンを試みるとを調査します。  

-   自体が誰かが追加、DA グループにメンバーがあるない場合は、すぐにチェックします。  

2 番目のイベントの例を示します。  

失敗したログオンの異常の数は、パスワード推測攻撃である可能性があります。 失敗したログオンの異常に多数のアラートを提供するエンタープライズ、悪意のあるセキュリティ イベントの前に、環境内で失敗したログオンの通常のレベルまず理解しなければなりません。  

セキュリティ侵害の兆候を監視する場合に含める必要があるイベントの包括的なについてを参照してください[付録 l:監視するイベント](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md)します。

-   アカウントを有効にするための組み込み管理者アカウントの Active Directory とメンバー システム) など特権アカウントを無効になっています。

## <a name="general-list-of-security-event-id-recommendation-criticalities"></a>セキュリティ イベント ID の推奨事項 Criticalities の [全般] の一覧  
すべてのイベント ID の推奨事項は、次のように評価の重要度は含まれます。  

**高。** 高重要度評価でイベント Id が常にすぐに警告を表示および調査。  

**[中]:** 中規模の重要度評価でイベント ID は、悪意あるアクティビティである可能性がありますが、その他のいくつか異常が伴う必要があります (たとえばに特定の期間、予期しない出現回数、またはコンピューターで発生するで発生している異常な番号を通常を期待できませんイベント ログに記録します。)。 重要度のメディア イベントの場合も r としてメトリックを収集し、時間の経過と共にと比較します。  

**低:** 重要度の低いイベントでイベント ID が注意を凌いでいない、または中または高重要度のイベントと相関関係しない限り、アラートが発生する必要があります。  

これらの推奨事項については、管理者のベースラインのガイドを提供するものです。 すべての推奨事項は、運用環境で実装する前に徹底的にレビューする必要があります。  

参照してください[付録 l:監視するイベント](../../../ad-ds/plan/Appendix-L--Events-to-Monitor.md)モニター、その重要度評価、およびイベント メッセージの概要に推奨されるイベントの一覧についてはします。
