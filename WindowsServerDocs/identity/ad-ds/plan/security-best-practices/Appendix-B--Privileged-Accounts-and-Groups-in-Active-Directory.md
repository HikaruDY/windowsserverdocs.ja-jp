### <a name="built-in-privileged-accounts-and-groups"></a>組み込み特権アカウントとグループ  
Active Directory の目的は、管理の委任および権限とアクセス許可を割り当てるときに最低限の特権の原則を容易にします。 Active Directory ドメインにアカウントを持っている"regular"のユーザーがディレクトリに格納される内容の多くを読めるように、既定しますが、ディレクトリ内のデータの非常に限定されたセットのみを変更することは。 追加の特権を必要とするユーザーには、各自の役割に関連する特定のタスクを実行できますが、各自の作業に関係のないタスクを実行することはできません、ようにディレクトリに組み込まれているさまざまな権限を持つグループのメンバーシップを付与できます。  

Active Directory 内では、ディレクトリ内の最上位の特権グループを構成する 3 つの組み込みグループ: Enterprise Admins (EA) グループ、ドメイン管理者 (DA) グループ、および組み込みの管理者 (BA) グループ。  

4 番目のグループ、スキーマ管理者 (SA) のグループが特権を悪用している場合を破損または破壊 Active Directory フォレスト全体ですがこのグループより限定的にその機能で EA、DA、および BA よりもグループ。  

これら 4 つのグループだけでなく、さまざまな追加の組み込みと既定のアカウントと権限と実行する特定の管理タスクを許可するアクセス許可の付与は、Active Directory のグループがあります。 この付録の内容から、Active Directory のすべての組み込みまたは既定のグループの詳細な説明が提供されないグループと、インストールで最もよく見いるアカウントのテーブルは提供します。  

たとえば、Active Directory フォレストに Microsoft Exchange Server をインストールする場合追加のアカウントとグループ可能性があります作成、ドメイン内の組み込みとユーザーのコンテナーで。 この付録では、グループおよびネイティブの役割と機能に基づいて、Active Directory の組み込みとユーザーのコンテナー内で作成したアカウントのみについて説明します。 アカウントとエンタープライズ ソフトウェアのインストールによって作成されるグループは含まれません。

Enterprise Admins (EA) グループは、フォレスト ルート ドメインにあるし、既定では、フォレスト内の各ドメインの組み込みの Administrators グループのメンバーであります。 フォレスト ルート ドメインのビルトイン Administrator アカウントは、EA のグループの唯一の既定のメンバーです。 EAs 権限およびフォレスト全体の変更に影響することを許可するアクセス許可が付与されます。 これらは、追加、ドメインを削除するまたはフォレストの信頼を確立する、またはフォレスト機能レベルを上げるなど、フォレスト内のすべてのドメインに影響する変更です。 正しく設計と実装の委任モデルでは、まずフォレストを構築するときにのみ、または送信フォレスト信頼を確立するなど特定のフォレスト全体の変更を行うときにで EA のメンバーシップが必要です。  

EA のグループが既定では、フォレスト ルート ドメインのユーザー コンテナー内にあるし、フォレスト ルート ドメインは、グループの場合は、グローバル セキュリティ グループの Windows 2000 Server 混在モードで実行されていない限り、ユニバーサル セキュリティ グループとは。 直接 EA グループにいくつかの権限の付与がフォレスト内の各ドメインの Administrators グループのメンバーであるため、EA グループによって実際に継承多くをこのグループの権限のされます。 Enterprise Admins は、ワークステーションまたはメンバー サーバーの既定の権限を持っているありません。  

#### <a name="domain-admins"></a>Domain Admins  
フォレスト内の各ドメインが、独自のドメイン管理者 (DA) グループ、ドメインに参加しているすべてのコンピューターでローカルの Administrators グループのメンバーだけでなく、そのドメインの組み込みの管理者 (BA) グループのメンバーであります。 DA、ドメイン グループの唯一の既定のメンバーは、そのドメインのビルトイン Administrator アカウントです。  

DAs は、EAs がフォレスト全体の特権のドメイン内で強力なです。 正しく設計と実装の委任モデルでは、ドメイン内のすべてのコンピューターに対する高いレベルの特権を持つアカウントに必要なまたはときに特定の状況が「非常時」のシナリオでのみ DA メンバーシップを必要するドメイン全体変更を行う必要があります。 ネイティブ Active Directory 委任メカニズムはできますが、委任できる程度に緊急の場合にのみ DA アカウントを使用することは、時間がかかり、することができます、効果的な委任モデルを構築して、多くの組織を使用して、サード パーティ製アプリケーションを迅速に処理をします。
