# <a name="planning-operations-master-role-placement"></a>操作マスターの役割の配置を計画します。

>適用対象: Windows Server 2016、Windows Server 2012 R2、Windows Server 2012

Active Directory ドメイン サービス (AD DS) には、任意のドメイン コントローラーがディレクトリの変更をそのまま使用し、変更を他のすべてのドメイン コントローラーにレプリケートを意味のディレクトリ データのマルチ マスター レプリケーションがサポートしています。 ただし、スキーマの変更など、特定の変更は、マルチ マスター方法で実行するが難しいではありません。 このため、操作マスターと呼ばれる特定ドメイン コントローラーの役割を保有特定の特定の変更要求の受け入れを担当します。  
  
> [!NOTE]  
> 操作マスターの役割所有者は、いくつかの情報を Active Directory データベースに書き込むことである必要があります。 読み取り専用の性質読み取り専用ドメイン コントローラー (RODC) で Active Directory データベース、Rodc は、操作マスターの役割所有者として機能できません。  
  
次の 3 つの操作マスターの役割 (とも呼ばれるフレキシブル シングル マスター操作または FSMO) は、各ドメインに存在します。  
  
-   プライマリ ドメイン コントローラー (PDC) エミュレータ操作マスタは、すべてのパスワードの更新を処理します。  
  
-   相対 ID (RID) 操作マスタは、ドメインのグローバル RID プールを維持し、ローカル Rid プールのドメインで作成されたすべてのセキュリティ プリンシパルの一意識別子であることを確認するすべてのドメイン コントローラーを割り当てます。  
  
-   特定のドメインのインフラストラクチャ操作マスタは、そのドメイン内のグループのメンバーである他のドメインのセキュリティ プリンシパルの一覧を保持します。  
  
に加えて、次の 3 つのドメイン レベルの操作マスターの役割は、次の 2 つの操作マスターの役割は、各フォレスト内があります。  
  
-   スキーマ操作マスタは、スキーマの変更を制御します。  
  
-   ドメイン名前付け操作マスタは、追加し、フォレストとの間にドメインと他のディレクトリ パーティション (たとえば、ドメイン ネーム システム (DNS) のアプリケーション パーティション) を削除します。  
  
ネットワークの信頼性が高く、領域の操作マスターの役割をホストしているドメイン コントローラーを配置し、PDC エミュレーターと RID マスターが常に使用可能であることを確認します。  
  
操作マスターの役割所有者は、指定したドメイン内の最初のドメイン コントローラーの作成時に自動的に割り当てられます。 次の 2 つのフォレスト レベルのロール (スキーマ マスターとドメイン名前付けマスター) は、フォレスト内に作成された最初のドメイン コントローラーに割り当てられます。 さらに、次の 3 つのドメイン レベルのロール (RID マスタ、インフラストラクチャ マスター、および PDC エミュレーター) は、ドメインで作成した最初のドメイン コントローラーに割り当てられます。  
  
> [!NOTE]  
> 新しいドメインを作成するときにのみ、および現在の役割所有者の降格時に自動の操作マスターの役割所有者の割り当てが行われます。 ロールの所有者にその他のすべての変更は、管理者によって開始する必要があります。  
  
これらの自動操作マスターの役割の割り当てには、フォレストまたはドメインで作成された最初のドメイン コントローラー上の CPU 使用率が非常に高い可能性があります。 これを避けるために (転送) 操作マスターの役割を割り当てる、フォレストまたはドメイン内のさまざまなドメイン コントローラー。 ホスト操作マスターのネットワークの信頼性の高い場所と、フォレスト内にあるその他のすべてのドメイン コントローラーで、操作マスタにアクセスできる場所の領域における役割、ドメイン コントローラーを配置します。  
  
スタンバイ (代替) の操作を指定する必要がありますもマスターすべての操作マスターの役割。 スタンバイの操作マスタは、ドメイン コントローラーの元の役割所有者が失敗する場合にする操作マスターの役割を転送する可能性があります。 スタンバイ操作マスターが、実際の操作マスタの直接のレプリケーション パートナーであることを確認します。  
  
## <a name="planning-the-pdc-emulator-placement"></a>PDC エミュレーターの配置を計画します。  
PDC エミュレーターは、クライアントのパスワードの変更を処理します。 1 つだけのドメイン コントローラーはフォレスト内の各ドメインの PDC エミュレーターとして機能します。  
  
場合でも、すべてのドメイン コントローラーが Windows 2000、Windows Server 2003、および Windows Server 2008 にアップグレードして、ドメインが Windows 2000 ネイティブの機能レベルで動作している、PDC エミュレーターは、ドメイン内の他のドメイン コントローラーで実行されるパスワード変更の優先的なレプリケーションを受信します。 パスワードが最近変更された場合、その変更は、ドメイン内のすべてのドメイン コントローラーにレプリケートされる時間をかかります。 パスワードが間違ってのための別のドメイン コントローラーでログオン認証に失敗した場合、そのドメイン コントローラーでは、PDC エミュレーターに認証要求を受理またはログオンを拒否するかどうかを決定する前に転送されます。  
  
PDC エミュレーターを多数のパスワードが必要な場合は、操作を転送用には、そのドメインからユーザーを含む場所に配置します。 さらに、場所がレプリケーションの待機時間を最小限に抑えるには、他の場所にも接続されていることを確認します。

## <a name="requirements-for-infrastructure-master-placement"></a>インフラストラクチャ マスターの配置の要件  

インフラストラクチャ マスターでは、独自のドメイン内のグループに追加されている他のドメインのセキュリティ プリンシパルの名前を更新します。 など、ユーザーが 1 つのドメインから別のドメイン内のグループのメンバーである最初のドメインにユーザーの名前が変更された場合は、2 つ目のドメインには通知されず、グループのメンバーップ一覧で、ユーザーの名前を更新する必要があります。 1 つのドメインのドメイン コントローラーはセキュリティ プリンシパルが別のドメイン内のドメイン コントローラーにレプリケートしない、ため 2 つ目のドメインにない場合は、インフラストラクチャ マスターの変更を認識なりません。  
  
インフラストラクチャ マスター常にモニター グループ メンバーシップを他のドメインのセキュリティ プリンシパルを探しています。 いずれかが見つかると、情報が更新されていることを確認するセキュリティ プリンシパルのドメインを確認します。 情報が有効期限切れの場合は、インフラストラクチャ マスターが、更新プログラムを実行し、そのドメイン内の他のドメイン コントローラに変更をレプリケートします。  
  
2 つの例外は、このルールに適用されます。 最初に、[すべてのドメイン コントローラーがグローバル カタログ サーバーの場合は、インフラストラクチャ マスターの役割をホストするドメイン コントローラーはグローバル カタログが属するドメインに関係なく、更新された情報をレプリケートするので重要です。 2 番目に、[1 つのドメイン、フォレストの場合は、インフラストラクチャ マスターの役割をホストするドメイン コントローラーは他のドメインのセキュリティ プリンシパルが存在しないため重要です。
