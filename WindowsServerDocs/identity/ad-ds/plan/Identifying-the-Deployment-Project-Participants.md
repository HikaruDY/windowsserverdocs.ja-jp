# <a name="identifying-the-deployment-project-participants"></a>展開プロジェクトの参加者を識別します。

## <a name="BKMK_1"></a>プロジェクトに固有の役割を定義します。  
プロジェクト チームを確立する重要な手順では、プロジェクトに固有の役割を保持する個人を特定します。 スポンサー、プロジェクト設計者および管理者が含まれます。 これらの担当者は、Active Directory 展開プロジェクトを実行する責任者です。  
  
プロジェクト兼プロジェクト マネージャを指名する後ような専門家は、組織全体での通信チャネルを確立プロジェクトのスケジュールを作成し、以降では、さまざまな所有者は、プロジェクト チームのメンバーとなる個人を特定します。  
  
### <a name="executive-sponsor"></a>スポンサー  
AD DS などのインフラストラクチャを展開する組織に広範な影響を受けることができます。 このために重要な展開のビジネス価値を理解、executive レベルでは、プロジェクトをサポートし、組織全体の競合の解決に役立つエグゼクティブ スポンサーです。  
  
### <a name="project-architect"></a>プロジェクト アーキテクト  
各 Active Directory 展開プロジェクトでは、Active Directory の設計と展開の意思決定プロセスを管理するプロジェクトの設計者が必要です。 設計者は、設計と AD DS の展開のプロセスを支援する技術的な専門知識を提供します。  
  
> [!NOTE]  
> 組織内の既存の担当者には、directory 設計のエクスペリエンスがない、ユーザーは Active Directory の設計と展開の専門家社外コンサルタントを雇ってする可能性があります。  
  
Active Directory プロジェクトの設計者の役割は次のとおりです。  
  
-   Active Directory の設計を所有しています。  
  
-   設計上の決定を理解し、キーの論理的根拠を記録します。  
  
-   設計が、組織のビジネス ニーズを満たしていることを確認  
  
-   設計、展開、および運用チームの間で連携してコンセンサス ・を確立します。  
  
-   AD DS 統合アプリケーションのニーズを理解します。  
  
最終的な Active Directory 設計では、ビジネスの目標と技術上の決定の組み合わせを反映する必要があります。 そのため、プロジェクトの設計者は、ビジネスの目標と整合していることを確認する設計に関する決定事項を確認する必要があります。  
  
### <a name="project-manager"></a>プロジェクト マネージャ  
プロジェクト マネージャには、事業単位間およびテクノロジの管理グループ間で連携が容易になります。 理想的には、Active Directory 展開プロジェクト マネージャでは、IT グループと AD DS の展開を準備しているグループ用の設計要件のある、運用、両方のポリシーを理解している組織内でユーザーからです。 プロジェクト マネージャでは、設計で始まると、実装、までに、展開全体プロジェクトを監視し、スケジュールと予算内で、プロジェクトが残ることを確認します。 プロジェクト マネージャの役割は次のとおりです。  
  
-   基本的なプロジェクトのスケジューリングや予算などの計画を提供します。  
  
-   Active Directory の設計と展開プロジェクトの進行状況を促進  
  
-   適切なユーザーが設計プロセスの各部分に関係していることを確認  
  
-   単一の Active Directory 展開プロジェクトの接続ポイントとして機能してください。  
  
-   設計、展開、および運用チームの間の通信を確立します。  
  
-   確立して、展開プロジェクト全体にわたるスポンサーとの通信を維持します。  
  
## <a name="BKMK_2"></a>所有者および管理者を確立します。  
Active Directory 展開プロジェクトでは、所有者にいるユーザーはによって保持されている責任を管理すると、その展開のタスクを完了し、その Active Directory デザインの仕様は、組織のニーズを満たすことを確認します。 所有者は、必ずしもへのアクセスまたはディレクトリ インフラストラクチャを直接操作しないでください。 管理者は、必須の展開のタスクを完了するための担当者です。 管理者は、ネットワーク アクセス権と、ディレクトリとそのインフラストラクチャを操作するために必要なアクセス許可があります。  
  
所有者の役割は、戦略的な経営上です。 所有者は、フォレスト内の新しいドメイン コント ローラーの作成など、Active Directory 設計の実装のために必要なタスクに管理者に通信するためです。 管理者は、設計の仕様に従ってネットワークの設計を実装する責任者です。  
  
大規模な組織でさまざまな個人が所有者と代理管理者の役割を埋めるただし、一部小規模な組織では、同じ個人は所有者と、管理者の両方として動作可能性があります。  
  
### <a name="service-and-data-owners"></a>サービスとデータの所有者  
AD DS が日常的に管理するには、所有者の 2 種類が含まれます。  
  
-   Active Directory インフラストラクチャの計画と長期的なメンテナンスのため、ディレクトリが引き続き機能し、サービス レベル アグリーメントに確立された目標が維持されるようにすることの責任を負うサービスの所有者  
  
-   データ所有者がディレクトリに保存された情報の保守を担当します。 これには、ユーザーとコンピューター アカウントの管理とメンバー サーバーおよびワークステーションなどのローカル リソースの管理が含まれます。  
  
ことが重要としてできるだけの設計プロセスの多くに参加できるように初期 Active Directory サービスとデータ所有者を特定します。 展開プロジェクトが完了したら、サービスとデータ所有者はディレクトリの長期的な保守を担当する、ために、これらの個人に関する組織のニーズに入力して、特定の設計に関する決定事項が行われる方法と理由を理解するため重要です。 サービスの所有者には、フォレストの所有者、Active Directory ドメイン ネーム システム (DNS) の所有者、およびサイトのトポロジ所有者が含まれます。 データ所有者には、組織単位 (OU) の所有者が含まれます。  
  
### <a name="service-and-data-administrators"></a>サービスとデータの管理者  
AD DS の操作は、次の 2 つの種類の管理者: サービス管理者とデータ管理者。 サービス管理者は、サービスの所有者によってなされた決定をポリシーを実装し、日常的なディレクトリ サービスおよびインフラストラクチャの管理に関連するタスクを処理します。 これは、AD DS フォレスト全体の設定の構成を制御して、ディレクトリが常に利用できることを保証するために必要な DNS などの他のネットワーク サービスの管理、ディレクトリ サービスをホストしているドメイン コント ローラーの管理が含まれます。  
  
サービス管理者は、最初の Windows Server 2008 の Active Directory 展開プロセスが完了した後に必要な継続的な Active Directory 展開タスクを完了する担当もあります。 たとえば、ディレクトリに増加要求に対応する、サービス管理者追加のドメイン コント ローラーの作成しの確立または必要に応じて、ドメイン間の信頼関係を削除します。 このため、Active Directory 展開チームをサービス管理者を含める必要があります。  
  
組織内の信頼されたユーザーにのみ、サービス管理者の役割を割り当てるように注意する必要があります。 これらの担当者にドメイン コント ローラーのシステム ファイルを変更する機能があるために、AD DS の動作を変更できます。 組織内のサービス管理者は、運用についてよく知っている個人およびこれらのポリシーを適用する必要性を理解しし、ネットワーク上の場所になっているセキュリティ ポリシー、することを確認する必要があります。  
  
データ管理者は、ユーザー アカウントとグループ アカウントなどの AD DS に格納されているデータを維持するためと、そのドメインのメンバーであるコンピューターを維持するための両方を担当するドメイン内のユーザー。 データの管理者は、ディレクトリ内のオブジェクトのサブセットが制御され、インストールや、ディレクトリ サービスの構成を制御はありません。  
  
データ管理者アカウントが、既定では提供されません。 設計チームと判断組織の管理対象リソースが、ドメインの所有者必要がありますデータ管理者アカウントを作成し、それらを管理者が担当するオブジェクトのセットに基づいて適切なアクセス許可を委任します。  
  
インフラストラクチャが機能し続けるために必要な最小数を組織内のサービス管理者の数を制限することをお勧めします。 データ管理者によっては、大部分の管理作業を完了できます。 サービス管理者が必要量より幅の広いスキル ディレクトリとそれをサポートするインフラストラクチャを維持する必要があるためです。 データの管理者には、ディレクトリの一部を管理するために必要なスキルのみが必要です。 この方法での作業の割り当てを分割結果、組織のコストを削減のため、管理者の数が少ないが動作し、ディレクトリ全体とインフラストラクチャの管理のためのトレーニングする必要があります。  
  
たとえば、サービス管理者は、フォレストにドメインを追加する方法を理解する必要があります。 これには、サーバーをドメイン コント ローラーに変換するソフトウェアをインストールする方法と、ドメイン コント ローラーは、Active Directory 環境にシームレスにマージできますできるように DNS 環境を操作する方法が含まれます。 のみ、データは、管理者は、部門の新しい従業員用の新しいユーザー アカウントの作成などを担当している特定のデータを管理する方法を理解する必要があります。  
  
AD DS の展開には、ネットワーク インフラストラクチャの操作に関係する多数の異なるグループ間の通信で調整する必要があります。 これらのグループには、設計と展開プロセス中に、さまざまなグループを表すを担当しているサービスとデータの所有者を指名する必要があります。  
  
展開プロジェクトが完了すると、そのグループによって管理インフラストラクチャの部分を担当するこれらのサービスとデータ所有者が続行します。 Active Directory 環境でこれらの所有者は、フォレストの所有者、AD DS の所有者の DNS、サイトのトポロジ所有者、および OU の所有者です。 これらのサービスとデータ所有者の役割は、次のセクションで説明します。  
  
#### <a name="forest-owner"></a>フォレストの所有者  
フォレストの所有者は、Active Directory の展開プロセスを担当し、だれが最終的に責任を持つ、展開が完了した後に、フォレスト内でのサービス提供を維持するために、組織内の上級情報技術 (IT) マネージャーでは通常です。 フォレストの所有者は、ネットワーク インフラストラクチャと管理上のニーズについて必要な情報を提供するには、組織内でキーの担当者を識別することによって、その他の役割の所有権を格納する個人を割り当てます。 フォレストの所有者は、次を担当します。  
  
-   フォレストを作成するフォレスト ルート ドメインの展開  
  
-   フォレストに必要なドメインを作成するには、各ドメインの最初のドメイン コント ローラーの展開  
  
-   フォレストのすべてのドメイン内のサービス管理者グループのメンバーシップ  
  
-   フォレスト内の各ドメインの OU 構造の設計の作成  
  
-   OU の所有者に管理権限の委任  
  
-   スキーマの変更  
  
-   フォレスト全体の構成設定を変更する  
  
-   詳細なパスワードとアカウント ロックアウトのポリシーなどのドメイン ユーザー アカウント ポリシーを含む、特定のグループ ポリシー ポリシー設定の実装  
  
-   ドメイン コント ローラーに適用されるビジネス ポリシーの設定  
  
-   ドメイン レベルで適用されているその他のグループ ポリシー設定  
  
フォレストの所有者は、フォレスト全体で権限を持ちます。 フォレストの所有者の責任をグループ ポリシーとビジネス ポリシーを設定して、サービス管理者にいるユーザーを選択することをお勧めします。 フォレストの所有者は、サービスの所有者です。  
  
#### <a name="dns-for-ad-ds-owner"></a>AD DS の所有者の DNS  
AD DS の所有者の DNS は、既存の DNS インフラストラクチャをよく理解し、組織の既存の名前空間を持つ個人です。  
  
AD DS の所有者の DNS は、次を担当します。  
  
-   設計チームと、DNS インフラストラクチャを現在所有している IT グループ間での担当者に提供しています。  
  
-   新しい Active Directory 名前空間の作成を支援する組織の既存の DNS 名前空間に関する情報を提供します。  
  
-   設計チームの仕様に従って、新しい DNS インフラストラクチャが展開されていること、および正しく機能していることを確認して、展開チームの作業  
  
-   DNS サーバー サービスと DNS データを含む、AD DS インフラストラクチャ用の DNS を管理します。  
  
AD DS の所有者の DNS は、サービスの所有者です。  
  
#### <a name="site-topology-owner"></a>サイト トポロジの所有者  
サイトのトポロジ所有者は、個別のサブネット、ルーター、および低速リンクによって接続されているネットワークの領域のマッピングなど、組織のネットワークの物理構造を理解しています。 サイトのトポロジ所有者は、以下に、次を行います。  
  
-   物理ネットワーク トポロジと AD DS にどのように影響を理解します。  
  
-   Active Directory の展開が、ネットワークをどのように影響するかを理解します。  
  
-   作成する必要がある Active Directory 論理サイトを決定します。  
  
-   サブネットを追加すると、ドメイン コント ローラーのサイト オブジェクトを更新、変更、または削除  
  
-   サイト リンク、サイト リンク ブリッジ、および手動接続オブジェクトを作成します。  
  
サイトのトポロジ所有者は、サービスの所有者です。  
  
#### <a name="ou-owner"></a>組織単位の所有者  
OU の所有者は、ディレクトリに保存されているデータの管理を担当します。 この個人は、運用に精通していると、ネットワーク上に配置されているセキュリティ ポリシーである必要があります。 これらのタスクのみが割り当てられている Ou で実行できるおよび OU の所有者をサービス管理者に委任されたタスクのみを実行できます。 OU の所有者に割り当てられる可能性があるタスクを以下に示します。  
  
-   割り当てられたその OU 内のすべてのアカウント管理タスクの実行  
  
-   ワークステーションと、割り当てられた OU のメンバーであるメンバー サーバーの管理  
  
-   割り当てられたその OU 内のローカル管理者権限の委任  
  
OU の所有者は、データ所有者です。  
  
## <a name="BKMK_3"></a>建物プロジェクト チーム  
Active Directory プロジェクト チームは、Active Directory の設計と展開のタスクを完了するを担当する一時的なグループです。 Active Directory 展開プロジェクトが完了したら、所有者が、ディレクトリの責任を負うものし、プロジェクト チームを解除できます。  
  
プロジェクト チームのサイズは、組織の規模に応じて変化します。 小規模な組織では、1 人のユーザーはプロジェクト チームの責任の複数の領域をカバーし、展開の 1 つ以上のフェーズに含まれているできます。 大規模な組織は、さまざまな個人に大規模なチームまたは異なるチームの責任のさまざまな分野に対応する必要があります。 責任のすべての領域が割り当てられており、組織の設計目標を満たす限り、チームのサイズは重要ではありません。  
  
### <a name="identifying-potential-forest-owners"></a>潜在的なフォレストの所有者を識別します。  
組織内の所有およびディレクトリ サービスをネットワーク上のユーザーに提供するために必要なリソースを制御するグループを識別します。 これらのグループは、潜在的なフォレストの所有者と見なされます。  
  
AD DS のサービスとデータの管理の分離により、IT グループ (またはグループ) 組織のローカルの administrators グループごとに独自のグループに属しているデータを管理するときに、ディレクトリ サービスを管理するインフラストラクチャのできます。 潜在的なフォレストの所有者では、展開および AD DS をサポートするネットワーク インフラストラクチャ上で、必要な権限があります。  
  
IT 部署が 1 つ一元的なインフラストラクチャがある組織では、IT グループは、通常、フォレストの所有者と、潜在的なフォレストの所有者、将来の展開 番号が含まれている組織の IT の独立したインフラストラクチャのグループにある潜在的なフォレストの所有者の数。 組織は既にか所で Active Directory インフラストラクチャには場合、現在のフォレスト所有者も新しい展開の潜在的なフォレストの所有者できます。  
  
展開を検討している各フォレスト、フォレストの所有者として動作する潜在的なフォレストの所有者のいずれかを選択します。 これらの潜在的なフォレスト所有者は、フォレストの実際に展開するかどうか、または代替の一連の (別の既存のフォレストに参加) などの操作が使用可能なリソースを活用しもが自分のニーズを満たしているかどうかを決定する設計チームとします。 組織内、フォレストの所有者 (または所有者) は、Active Directory 設計チームのメンバーです。  
  
### <a name="establishing-a-design-team"></a>設計チームを確立します。  
Active Directory 設計チームは、Active Directory 論理構造の設計に関する決定を行うために必要なすべての情報を収集します。  
  
次のような設計チームの責任があります。  
  
-   フォレストとドメインの数が必要との関係はフォレストおよびドメインの間での決定  
  
-   データ所有者が設計が、セキュリティや管理要件を満たしていることを確認の操作  
  
-   確認し、現在のネットワーク インフラストラクチャの設計をサポートしていること、デザインが悪影響を及ぼさない、ネットワーク上に展開された既存のアプリケーションの現在のネットワーク管理者の操作  
  
-   組織のセキュリティ グループのデザインが確立されているセキュリティ ポリシーを満たしていることを確認する担当者の操作  
  
-   データ所有者に適切なレベルの保護と適切な権限の委任を許可する OU 構造の設計  
  
-   テストの設計、展開チームとラボでとして機能していることを確認するための環境の計画し、に発生した問題に対処するために必要なとして設計の変更  
  
-   利用可能な帯域幅のオーバー ロードを妨げているときに、フォレストのレプリケーション要件を満たしてサイト トポロジの設計を作成します。 サイト トポロジの設計に関する詳細については、次を参照してください。 [、サイトのトポロジの Windows Server 2008 AD DS の設計](https://technet.microsoft.com/library/cc772013.aspx)します。  
  
-   設計が正しく実装されていることを確認する展開チームの操作  
  
設計チームには、次のメンバーが含まれています。  
  
-   潜在的なフォレストの所有者  
  
-   プロジェクト アーキテクト  
  
-   プロジェクト マネージャ  
  
-   確立して、ネットワーク上のセキュリティ ポリシーの保守を担当します。  
  
論理構造の設計プロセス中には、設計チームは、他の所有者を特定します。 それらが識別されたとすぐには、設計プロセスに参加しているような専門家が開始する必要があります。 展開プロジェクト展開チームに渡されますが後、設計チームは、設計が正しく実装されていることを確認する、展開プロセスを監視します。 設計チームは、テストからのフィードバックに基づいて設計するもの変更を行います。  
  
### <a name="establishing-a-deployment-team"></a>展開チームを確立します。  
Active Directory 展開チームは、テスト、および Active Directory 論理構造の設計の実装を担当します。 これには、次のタスクが含まれます。  
  
-   十分に実稼働環境をエミュレートするテスト環境を確立します。  
  
-   各役割の所有者の目的を満たしていることを確認するラボ環境で提案されたフォレストとドメイン構造を実装することで、設計をテストします。  
  
-   開発およびテスト ラボ環境での設計を提案されたすべての移行シナリオ  
  
-   ある所有者ごとサインオフ テスト プロセスで正しいデザイン機能がテストされていることを確認することを確認  
  
-   パイロットの環境で展開操作をテストします。  
  
設計とテストのタスクが完了すると、展開チームは、次のタスクを実行します。  
  
-   Active Directory 論理構造の設計に従ってドメインおよびフォレストを作成します。  
  
-   サイトおよび必要に応じて、サイト リンク オブジェクトをサイト トポロジの設計にに基づいて作成します。  
  
-   により、AD DS をサポートするために、DNS インフラストラクチャが構成されているすべての新しい名前空間は、組織の既存の名前空間に統合されています。  
  
Active Directory 展開チームには、次のメンバーが含まれています。  
  
-   フォレストの所有者  
  
-   AD DS の所有者の DNS  
  
-   サイト トポロジの所有者  
  
-   OU の所有者  
  
展開チームは、運用チームのメンバーがこの新しい設計を理解していることを確認する展開フェーズ中に、サービスとデータの管理者とは動作します。 これにより、展開操作が完了すると、所有権のスムーズな移行をことを確認します。 展開プロセスが完成したら、新しい Active Directory 環境を維持するための責任は、運用チームに渡します。  
  
### <a name="documenting-the-design-and-deployment-teams"></a>設計と展開チームを文書化  
名前を文書化し、設計と AD DS の展開に参加するユーザーの情報にお問い合わせください。 設計と展開チームの各役割を担当するユーザーを識別します。 最初に、この一覧には、潜在的なフォレストの所有者、プロジェクト マネージャー、およびプロジェクトの設計者が含まれています。 展開するフォレストの数を決定するときは、追加のフォレストに対して新しい設計チームを作成する必要があります。 チームのメンバーシップを変更して、設計プロセス中にさまざまな Active Directory の所有者を特定するようのドキュメントを更新する必要があることに注意してください。 各フォレストの設計と展開チームを文書化するためのワークシートで、ダウンロード Job_Aids_Designing_and_Deploying_Directory_and_Security_Services.zip ([https://go.microsoft.com/fwlink/?LinkID=102558](https://go.microsoft.com/fwlink/?LinkID=102558))"の設計と展開チームの情報] を開きます(DSSLOGI_1.doc) します。
