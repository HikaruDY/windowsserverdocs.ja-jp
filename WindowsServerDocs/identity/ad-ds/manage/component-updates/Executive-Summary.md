>次のドキュメントでは、2013年で作成されておりを履歴目的でのみ提供されます。  現在このドキュメントを確認しているし、変更されることができます。  現在のベスト プラクティスは反映可能性があります。

このドキュメントには、攻撃を軽減するには、Active Directory とその対策の最も一般的な攻撃がについて説明しますが、回復の完全なセキュリティ侵害が発生した場合の推奨事項も含まれます。 Active Directory の完全なセキュリティ侵害が発生した場合に回復するだけのことを確認方法は、前に、侵害対策を準備します。

## <a name="reducing-the-active-directory-attack-surface"></a>Active Directory の攻撃を削減する  
このセクションでは、Active Directory のインストールの攻撃対象領域を減らすために技術的なコントロールについて説明します。 このセクションの内容とは、次の項目です。  
  
-   **特権アカウントと Active Directory でグループ**セクションは、最上位の特権アカウントと権限を持つアカウントを保護するメカニズムと Active Directory 内のグループについて説明します。 Active Directory 内で 3 つの組み込みグループいます (Enterprise Admins、Domain Admins、および管理者)、ディレクトリ内の最上位の特権グループの追加のグループおよびアカウントの数が保護することも必要があります。  
  
-   **最低限の特権の管理モデルを実装する**セクションの推奨事項を提供するだけでなく、高い特権を持つアカウントの使用を日常的な管理が提供するリスクを識別するにはそのリスクを軽減します。  
  
のみ、過剰な権限が侵害された環境で Active Directory で見つかりません。 組織の開発が必要以上の権限の許可を付与する習慣は通常、インフラストラクチャ全体にあります。  
  
-   Active directory  
  
-   メンバー サーバー  
  
-   ワークステーションで  
  
-   アプリケーションで  
  
-   データ リポジトリ  
  
-   **実装のセキュリティで保護された管理ホスト**セクションには、接続されているシステムと Active Directory の管理をサポートするために構成されているコンピューターである、セキュリティで保護された管理ホストがについて説明します。 これらのホストは専用の管理機能と、(Microsoft Office) などの電子メール アプリケーション、web ブラウザーや生産性ソフトウェアなどのソフトウェアを実行しないでください。  
  
このセクションで次に示します。

## <a name="planning-for-compromise"></a>侵害対策を計画する  
このセクションには、組織が起こる前に、セキュリティ侵害の準備に役立つ推奨事項、完全なセキュリティ侵害が発生しました、前に、セキュリティ侵害イベントを検出し、ケースでの対応や回復のガイドラインを提供する制御を実装が含まれていますディレクトリの完全なセキュリティ侵害は攻撃者によって実現されます。 このセクションの内容とは、次の項目です。  
  
-   **アプローチの再考**-基本原則と、組織が最も重要な資産を配置できるセキュリティで保護された環境を作成するためのガイドラインが含まれています。 次のガイドラインは次のとおりです。  
  
    -   分離して、重要な資産をセキュリティで保護するための原則を識別します。  
  
    -   制限付きリスク ベースの移行計画を定義します。  
  
    -   必要に応じて、"nonmigratory"の移行を利用します。  
  
    -   「創造的な破棄」を実装します。  
  
    -   レガシ システムとアプリケーションを分離します。  
  
    -   エンドユーザーのセキュリティを簡素化します。  
  
-   **セキュリティで保護された環境を維持する**-推奨事項概要だけでなく、有効なセキュリティを効果的なライフ サイクル管理の開発で使用するためのガイドラインとして使用するためのものにはが含まれています。 このセクションの内容とは、次の項目です。  
  
    -   **Active Directory のビジネスを中心としたセキュリティのベスト プラクティスを作成する**- を効果的にユーザー、データ、アプリケーションおよび Active Directory によって管理されるシステムのライフ サイクル管理、これらの原則に従ってください。  
  
        -   **Active Directory データにビジネスの所有権を割り当てる**-インフラストラクチャ コンポーネントの所有権を割り当てる IT は Active Directory ドメイン サービス (AD DS)、ビジネスをサポートするために新しい従業員は、新しいアプリケーションに追加されているデータと新しい情報リポジトリや指定のビジネス単位、ユーザー データと関連付けることが必要があります。  
  
        -   **Business-Driven ライフ サイクル管理の実装**-Active Directory 内のデータのライフ サイクル管理を実装する必要があります。  
  
        -   **すべての Active Directory データを分類**-ビジネス オーナーは、Active Directory 内のデータの分類を提供する必要があります。 データの分類モデル内で、次の Active Directory データの分類に含まれるなります。  
  
            -   **システム**-分類サーバー カタログの作成、そのオペレーティング システムの役割、および IT で実行されるアプリケーションとレコードのビジネス所有者。  
  
            -   **アプリケーション**-機能、ユーザー ベース、およびそのオペレーティング システムによって、アプリケーションを分類します。  
  
            -   **ユーザー** -攻撃者のターゲットとする最も可能性のある Active Directory のインストールでアカウントをタグ付けおよび監視する必要があります。
