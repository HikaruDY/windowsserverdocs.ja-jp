## <a name="ADATM_BM"></a>Active Directory 管理階層モデル
この階層モデルの目的は、環境 (階層 0) のフル コントロールと攻撃者が頻繁に侵害されるリスクが高いワークステーション資産の間でのバッファー ゾーンのセットを使用して ID システムを保護します。

![階層モデルの 3 つの層を示す図](../media/securing-privileged-access-reference-material/PAW_RM_Fig1.JPG)

階層モデルは 3 つのレベルで構成し、標準ユーザー アカウントの管理アカウントにはのみが含まれます。

-   **階層 0** -環境内のエンタープライズ ID を直接制御します。 階層 0 には、その中のアカウント、グループ、および Active Directory フォレスト、ドメイン、またはドメイン コントローラーの直接または間接的管理機能があるその他の資産およびすべてのアセットが含まれています。 すべての階層 0 資産のセキュリティの機密性は互いのコントロールはすべて効果的に相当します。

-   **階層 1** -エンタープライズ サーバーおよびアプリケーションを制御します。 階層 1 の資産には、サーバーのオペレーティング システム、クラウド サービス、およびエンタープライズ アプリケーションが含まれます。 階層 1 の管理者アカウントでは、これらの資産でホストされているビジネス価値のかなりの管理制御があります。 一般的なロールの例は、これらのオペレーティング システムのすべてのエンタープライズ サービスに影響する機能を管理しているサーバー管理者です。

-   **階層 2** -ユーザーのワークステーションとデバイスの制御。 階層 2 の管理者アカウントでは、ユーザーのワークステーションやデバイスでホストされているビジネス価値のかなりの管理制御があります。 例として、ヘルプ デスクやコンピューターは、ほとんどのユーザー データの整合性に影響が及ぶことができますのでに管理者をサポートします。

> [!NOTE]
> 層は、管理用の資産を保護するための基本的な優先順位付けメカニズムとしても機能しますが、任意の階層にあるすべての資産の制御と、攻撃者がほとんどまたはすべてのビジネス資産にアクセスできることを検討することが重要です。 基本的な優先順位付けメカニズムとして便利ですが、攻撃者の難易度とコストです。 攻撃者はすべての ID (階層 0) またはサーバーのフル コントロールを使用して操作を簡単にあり、それよりも、クラウド サービス (階層 1) 個々 のワークステーションやユーザーのデバイスを組織のデータを取得する (階層 2) にアクセスする必要があります。

## <a name="CSP_BM"></a>クリーン ソースの原則
クリーン ソースの原則には、同様に信頼できるセキュリティ保護されているオブジェクトとしてすべてのセキュリティの依存関係が必要です。

![どのオブジェクトを制御でのサブジェクトは、そのオブジェクトのセキュリティの依存関係を示す図](../media/securing-privileged-access-reference-material/PAW_RM_Fig5.JPG)

オブジェクトを制御しているサブジェクトは、そのオブジェクトのセキュリティの依存関係です。 敵対者は、ターゲット オブジェクトを効果的に制御では何も制御できますが、そのターゲット オブジェクトを制御することができます。 このため、必要があります、すべてセキュリティの依存関係の保証がオブジェクト自体の目的のセキュリティ レベル以上のことを確認します。

原則は簡単、これを適用する必要があります (オブジェクト) の関心のある資産の制御の関係を理解し、すべてのセキュリティ依存関係 (サブジェクト) を検出する依存関係の分析を実行します。

制御は推移的であるために、この原則は繰り返される再帰的にするがします。 例の場合、コントロールし、B が C を制御するため、A も直接制御します。

(C) を含む、攻撃者が侵入は、コントロール (B を含む)、およびすべて B すべてにアクセスを制御します。 この同じ例では、セキュリティの依存関係の言語を使用して、B と A の両方 C のセキュリティの依存関係、その保証レベルを使用して C の順序で C の必要な保証レベルでセキュリティで保護されます。

##### <a name="rare-use-emergency-procedures"></a>まれな使用緊急時の手順
推奨する運用方法は、次の基準をサポートする必要があります。

-   停止を迅速に解決できることを確認します。

-   必要に応じて、まれな高い特権を持つタスクを完了できることを確認します。

-   安全な手順は、資格情報や特権を保護するために使用することを確認します。

-   適切な追跡および承認のプロセスに従うことを確認します。

## <a name="ESAE_BM"></a>ESAE 管理フォレスト設計のアプローチ
このセクションでは、サイバー セキュリティ攻撃からユーザーを保護する Microsoft のサイバー セキュリティ プロフェッショナル サービス チームによって展開されている、Enhanced Security Administrative Environment (ESAE) 参照アーキテクチャに基づく管理フォレストに対するアプローチについて説明します。

専用管理フォレストは運用環境よりも強力なセキュリティ制御が存在する環境で組織はホストの管理アカウント、ワークステーション、およびグループを使用できます。

このアーキテクチャでは、さまざまなアーキテクチャでは、単一のフォレスト、Privileged Access Workstation (Paw) を使用して管理ものであっても不可能であるか、または簡単に構成されていないセキュリティ制御を使用します。 これにより、標準、権限のないユーザーとして管理フォレストでは、実稼働環境で高度な権限をより優れた技術ガバナンスのアカウントをプロビジョニングできます。 このアーキテクチャを制限するための手段として、信頼の認証の選択の機能の使用もできるようにするだけでログオン (と資格情報の公開) の承認済みホストします。 高いレベルの保証が完全な再構築の複雑性とコストをかけずに運用フォレストに必要な場合、管理フォレストを運用環境の保証レベルを高める環境を提供できます。

このアプローチでは、Active Directory 環境にフォレストを追加、中に、コストと複雑さは、固定の設計、小規模なハードウェア/ソフトウェア フット プリント、少数のユーザーによって制限されます。

> [!NOTE]
> このアプローチは Active Directory の管理に適してしますが、多くのアプリケーションは標準的な信頼を使用する外部フォレストからアカウントで管理されていると互換性のないです。

管理フォレスト設計では、次の考慮事項を含める必要があります。

-   **制限された範囲**-管理フォレストの主な価値は、高いレベルのセキュリティ保証と、攻撃される残存リスクが低いでします。 他の管理機能とアプリケーションを格納するフォレストを使用できますが、それぞれの増加のスコープ内、フォレストとそのリソースに攻撃される面が大きくなります。 目的は、機能を制限内、フォレストと管理者のユーザーの攻撃対象領域を最小限に抑える、保持するため、各範囲を広げるを慎重に考慮する必要があります。

-   **信頼の構成**-管理フォレストへの管理対象 forestss またはドメインから信頼を構成します。

    -   一方向の信頼は、実稼働環境から管理フォレスト必要があります。 ドメイン信頼またはフォレストの信頼を使用できます。 管理フォレストのドメイン/が追加のアプリケーションは、双方向信頼関係、セキュリティ検証にする必要がありますが、Active Directory を管理するために管理対象ドメイン/フォレストを信頼する必要はありませんし、テストします。

    -   適切な運用ホストへのログオンのみに、管理フォレスト内のアカウントを制限する認証の選択を使用する必要があります。 ドメイン コントローラーを維持する権限を委任する Active Directory で、これ通常与える必要があります、「ログオン許可」に、管理フォレスト内の層 0 管理アカウントの指定されたドメイン コントローラーの右します。 詳細については、選択的な認証設定を構成するを参照してください。

-   **特権とドメインのセキュリティ強化**-管理フォレストは、最低限の特権 Active Directory 管理の要件に基づいてするように構成する必要があります。
