# <a name="authentication-policies-and-authentication-policy-silos"></a>認証ポリシーと認証ポリシー サイロ

>適用対象: Windows Server (半期チャネル)、Windows Server 2016

IT プロフェッショナル向けのこのトピックでは、認証ポリシー サイロおよびポリシー サイロに属するアカウントを制限することができますをについて説明します。 また、認証ポリシーを使用して、アカウントのスコープを制限する方法についても説明します。

認証ポリシー サイロおよびそれに付随するポリシーは、選択したユーザー、コンピューター、またはサービスに関連するだけであるシステムに高い特権を持つ資格情報を格納する方法を提供します。 サイロの定義し、Active Directory 管理センターと Active Directory Windows PowerShell コマンドレットを使用して、Active Directory ドメイン サービス (AD DS) で管理されていることができます。

認証ポリシー サイロは、コンテナーを管理者はユーザー アカウント、コンピューター アカウント、およびサービス アカウントを割り当てることができます。 アカウントの設定は、そのコンテナーに適用されている認証ポリシーで管理できます。 これにより、管理者が個々 のアカウントのリソースへのアクセスを追跡する必要があるなり、悪意のあるユーザーを他のリソースにアクセスする資格情報の盗難を防ぐのに役立ちます。

Windows Server 2012 R2 で導入された機能を使用すると、認証ポリシー サイロは、高い特権を持つユーザーのセットをホストを作成できます。 特権アカウントを使用して、ドメインの場所を制限する認証ポリシーをこのコンテナーを割り当てることができます。 アカウントが Protected Users セキュリティ グループの場合、Kerberos プロトコルの排他的使用など、追加の制御が適用されます。

これらの機能により、価値の高いホストに高価値アカウントの使用量を制限できます。 たとえば、エンタープライズ、スキーマ、およびドメイン管理者を含む新しいフォレスト管理者サイロを作成できます。 パスワードとドメイン コントローラーやドメイン管理者コンソール以外のシステムからスマート カード ベースの認証は失敗できるように、認証ポリシー サイロを構成できます。

認証ポリシー サイロと認証ポリシーの構成については、次を参照してください。[保護されるアカウントの構成方法](how-to-configure-protected-accounts.md)します。

### <a name="about-authentication-policies"></a>認証ポリシーについて
認証ポリシーは、Kerberos プロトコルのチケット保証チケット (TGT) 有効期間プロパティとアカウントの種類の認証アクセス制御条件を定義します。 ポリシーの基盤であり、認証ポリシー サイロと呼ばれる AD DS コンテナを制御します。

認証ポリシーは、以下のいずれかを制御します。

-   アカウントに設定されている更新不可の TGT の有効期間。

-   この条件は、デバイス アカウント、パスワードまたは証明書を使用してログインするために満たす必要があります。

-   ユーザーとデバイスが、アカウントの一部として実行されているサービスへの認証に対応する必要がある条件。

Active Directory アカウントの種類は、呼び出し元の役割として、次のいずれかを決定します。

-   **ユーザー**

    ユーザーは、常に既定で NTLM を使用する認証の試行は拒否される Protected Users セキュリティ グループのメンバーである必要があります。

    設定するユーザー アカウントの TGT の有効期間を短くかユーザー アカウントがサインインできるデバイスを制限するポリシーを構成することができます。 認証ポリシーで、ユーザーとデバイスが、サービスへの認証に対応する必要がある条件を管理するには、豊富な式を構成できます。

    詳細については、次を参照してください。[Protected Users セキュリティ グループ](protected-users-security-group.md)します。

-   **サービス**

    スタンドアロンの管理されたサービス アカウント、グループ管理サービス アカウント、またはこれら 2 種類のアカウントが使用されるサービスから派生したカスタム アカウント オブジェクト。 ポリシーは、デバイスのアクセス制御条件は、管理されたサービス アカウントの資格情報を Active Directory ID で特定のデバイスに制限するために使用を設定できます。 サービスすべてに対する認証が失敗するため、Protected Users セキュリティ グループのメンバーことはありませんする必要があります。

-   **コンピューター**

    コンピューター アカウント オブジェクト、またはコンピューター アカウント オブジェクトから派生したカスタム アカウント オブジェクトが使用されます。 ポリシーは、アクセス制御条件のユーザーとデバイスのプロパティに基づいてアカウントに対する認証を許可するために必要なを設定できます。 コンピューターに対するすべての認証が失敗するため、Protected Users セキュリティ グループのメンバーことはありませんする必要があります。 既定では、NTLM 認証を使用する試行は拒否されます。 コンピューター アカウントの TGT 有効期間を構成できません必要があります。

> [!NOTE]
> アカウントの集合で、ポリシー、認証ポリシー サイロに関連付けなくても認証ポリシーを設定することができます。 保護する 1 つのアカウントがある場合は、この方法を使用できます。

### <a name="BKMK_HowRestrictingSignOn"></a>動作のユーザーのサインインに制限する方法
これらの認証ポリシーは、アカウントに適用される、ためサービスによって使用されるアカウントにも適用されます。 特定のホストにあるサービスに対するパスワードの使用量を制限する場合は、この設定は便利です。 たとえば、サービス アカウントのパスワードを Active Directory ドメイン サービスから取得するホストが許可されている場所が構成されて管理グループにします。 ただし、初期認証するには、すべてのホストからそのパスワードを使用します。 アクセス制御条件を適用すると、追加の保護レイヤーは、パスワード、パスワードを取得できるようにホストのセットだけに制限することで実現できます。

システム、ネットワーク サービス、またはその他のローカル サービス アイデンティティとして実行されるサービスは、ネットワーク サービスに接続するときに、ホストのコンピューター アカウントを使用します。 コンピューター アカウントを制限することはできません。 したがって、サービスはない Windows ホストのコンピューター アカウントを使用している場合でも制限することはできません。

特定のホストにユーザーのサインインを制限するには、ホストの ID を検証するドメイン コントローラーが必要です。 Kerberos 認証とともに Kerberos 防御 (これは、ダイナミック アクセス制御の一部である) を使用する場合、ユーザーの認証元となるホストの tgt キー配布センターが提供されます。 この防御された TGT の内容を使用すると、ホストが許可されているかどうかを決定するアクセス チェックを完了します。

ユーザーが Windows にサインインした場合、または既定では、アプリケーションは、資格情報プロンプトで、ドメイン資格情報を入力するとき、Windows は、ドメイン コントローラーに防御されていない AS-REQ を送信します。 ユーザーがサポートしていないコンピューターから要求を送信する場合は、Windows 7 または Windows Vista を実行しているコンピューターなど、防御要求は失敗します。

次に、プロセスについて説明します。

-   Windows Server 2012 R2 を実行しているドメインのドメイン コントローラーは、ユーザー アカウントのクエリを実行し、防御された要求を必要とする最初の認証を制限する認証ポリシーが構成されているかどうかを決定します。

-   ドメイン コントローラーには、要求は失敗します。

-   防御する必要があるために、ユーザーはサインイン プロセスを再試行する Kerberos 防御をサポートするために有効になっている Windows 8、Windows 8.1 を実行するコンピューターを使用してログインを試行できます。

-   Windows では、ドメインが Kerberos 防御をサポートしています、送信、防御された AS-REQ サインイン要求を再試行することを検出します。

-   ドメイン コントローラーは、要求の防御に使用された TGT の中に、構成済みのアクセス制御条件とクライアント オペレーティング システムのアイデンティティ情報を使用してはアクセス検査を実行します。

-   アクセス検査に失敗した場合、ドメイン コントローラーは、要求を拒否します。

オペレーティング システムでは、Kerberos 防御をサポートされている場合でもアクセス制御要件が適用でき、アクセスが許可される前に満たす必要があります。 ユーザーは、Windows にログインまたはアプリケーションの資格情報プロンプトで、ドメイン資格情報を入力します。 既定では、Windows は、ドメイン コントローラーを防御されていない AS-REQ を送信します。 ユーザーが Windows 8.1 または Windows 8 など、防御をサポートしているコンピューターから要求を送信する認証ポリシーが次のように評価されます。

1.  Windows Server 2012 R2 を実行しているドメインのドメイン コントローラーは、ユーザー アカウントのクエリを実行し、防御された要求を必要とする最初の認証を制限する認証ポリシーが構成されているかどうかを決定します。

2.  ドメイン コントローラーは、要求の防御に使用される TGT の中に、構成済みのアクセス制御条件とシステムのアイデンティティ情報を使用してはアクセス検査を実行します。 アクセスのチェックは成功します。

    > [!NOTE]
    > 従来のワークグループ制限が構成されている場合も満たす必要があります。

3.  ドメイン コントローラーは、防御された応答 (AS-担当) で応答し、認証が続行します。

### <a name="BKMK_HowRestrictingServiceTicket"></a>サービス チケット発行の制限のしくみ
アカウントが許可されていないと、TGT を持つユーザーは、サービスに接続しようとしています (など、サービスのサービス プリンシパル名 (SPN) によって識別されるサービスに対する認証を必要とするアプリケーションを開いて、次の順序が発生します。

1.  SPN から SPN1 への接続を試行する際、SPN1 へのサービス チケットを要求しているドメイン コントローラーに、TGS-REQ が送信されます。

2.  Windows Server 2012 R2 を実行しているドメインのドメイン コントローラーは spn1 をサービスの Active Directory ドメイン サービス アカウントを検索して、サービス チケット発行を制限する認証ポリシーを使用して、アカウントが構成されていることを決定します。

3.  ドメイン コントローラーで、構成済みのアクセス制御条件と、ユーザーの ID 情報 TGT を使用したはアクセス検査を実行します。 アクセスのチェックは失敗します。

4.  ドメイン コントローラーは、要求を拒否します。

アカウントが許可されているため、アカウントが、認証ポリシーによって設定されているアクセス制御条件を満たしているし、TGT を持つユーザーがサービスに接続しようと (など、認証を必要とするアプリケーションを開くことによってサービスが SPN で識別されるサービスに)、次の順序が発生します。

1.  SPN1 への接続を試行する際、SPN1 へのサービス チケットを要求しているドメイン コントローラーに、TGS-REQ が送信されます。

2.  Windows Server 2012 R2 を実行しているドメインのドメイン コントローラーは spn1 をサービスの Active Directory ドメイン サービス アカウントを検索して、サービス チケット発行を制限する認証ポリシーを使用して、アカウントが構成されていることを決定します。

3.  ドメイン コントローラーで、構成済みのアクセス制御条件と、ユーザーの ID 情報 TGT を使用したはアクセス検査を実行します。 アクセスのチェックは成功します。

4.  ドメイン コントローラーは、チケット保証サービス応答 (TGS-REP) で、要求に応答します。
